# Schema v2 (Phase 0) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the full Schema v2 migration described in
`docs/superpowers/specs/2026-06-23-bulletproof-v2-design.md`'s "Schema v2
(Phase 0 — full detail)" section: 4 new taxonomy tables, an expanded
`user_profile` and `exercises`, a simplified `session_type` enum, 4 new
tables (`recommendation_blocks`, `recommendation_block_exercises`,
`exercise_logs`, `daily_feedback`), `sessions` additions for the
start/end-workout flow, and real multi-user RLS (`owner_id` +
`auth.uid()` policies) everywhere — against the live, linked production
Supabase project, with the existing 1-row `user_profile`, 17-row
`exercises` seed, and 665/668/610/3-row `sessions`/`activity`/`recovery`/
`recommendations` history surviving every step intact.

**Architecture:** 12 new sequential SQL migration files under
`supabase/migrations/`, following this repo's existing one-file-per-
table/concern convention (see
`supabase/migrations/20260622000809_create_exercises.sql` +
`20260622000812_seed_exercises.sql` for the create/seed split, and
`20260622120000_add_authenticated_rls_policies.sql` for RLS as its own
later pass). No engine or app code changes — this phase is schema-only.
Each task pushes its migration to the linked remote project
(`npx supabase db push`) and verifies it with a REST API spot-check
before the next task starts.

**Tech Stack:** Postgres 17 (Supabase-managed), `supabase` CLI v2.107.0
(`npx supabase`, already authenticated via `SUPABASE_ACCESS_TOKEN` in the
repo-root `.env`), linked to project ref `gbtqzdjpkxpgkxjxrjoi`.

## Global Constraints

- **Every new column nullable or defaulted.** No exceptions — the
  existing 1-row `user_profile` and 17-row `exercises` seed must survive
  every migration in this plan with zero backfill blocking deploy.
- **Never use `ALTER TYPE ... DROP VALUE`.** It does not exist in
  Postgres. The `session_type` simplification (Task 7) uses the
  create-new-type → `CASE`-map data → drop old type → rename pattern.
- **The one real `auth.users` row is `89ac801b-5515-4388-80a7-03662030c487`**
  (Apple Sign-In, `smadimsetty@gmail.com`). Every `owner_id` backfill in
  this plan targets this literal UUID — confirmed via the Supabase Auth
  admin API before this plan was written. Do not invent or look up a
  different value.
- **The one real `user_profile` row is `id =
  'd8f1d832-d50b-4d23-962b-7272f78e2ede'`**, `name = 'Sohan Updated'`.
  Confirmed via direct REST API query before this plan was written.
- **Live row counts to verify against after every relevant task** (queried
  directly before this plan was written — re-query and compare, don't
  assume): `user_profile` 1, `exercises` 17, `recovery` 610, `activity`
  668, `sessions` 665, `recommendations` 3. `sessions.type` distribution:
  `rest` 496, `lower_a` 57, `upper_a` 50, `mobility` 30, `pickleball` 26,
  `run` 6 (note: `upper_b`/`lower_b` never actually appear in live data,
  but Task 7's `CASE` still maps them defensively). `recommendations.top_pick`/
  `runner_up` values seen: `lower_a`, `mobility`, `upper_a` only.
- **`.env` setup for every `npx supabase` command in this plan:** the
  repo-root `.env` (gitignored, already present in this worktree at
  `C:\Dev\bulletproof-schema-v2\.env`, copied from the main checkout)
  contains `SUPABASE_ACCESS_TOKEN`, `SUPABASE_URL`,
  `SUPABASE_SERVICE_ROLE_KEY`. Every CLI step in this plan runs as:
  ```bash
  set -a && source .env && set +a && export SUPABASE_ACCESS_TOKEN
  ```
  immediately before the `npx supabase` invocation, in the same shell
  command (the variable does not persist across separate tool calls).
  The project is already linked to ref `gbtqzdjpkxpgkxjxrjoi` in this
  worktree (`supabase/.temp/project-ref` already present) — do not
  re-run `supabase link` unless a step explicitly says to.
- **Every REST API verification query** uses:
  ```bash
  set -a && source .env && set +a
  curl -s "$SUPABASE_URL/rest/v1/<table>?select=<cols>" \
    -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" \
    -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"
  ```
  (the service role bypasses RLS, so this always reflects the table's
  true contents regardless of which policies a given task has or hasn't
  added yet).
- **Push one migration file at a time, verify, then proceed.** Never
  write multiple migration files and batch-push them untested — this
  phase touches substantial real production data (610+ rows in several
  tables) and every task's verification step must pass before the next
  task's migration file is even written.
- **No engine or app code changes anywhere in this plan.** If a task
  seems to require editing `engine/*.py` or `apps/*/lib/*.ts`, that's a
  signal something is wrong with the plan, not a reason to add one — flag
  it in the task's own notes instead (Task 8 does exactly this for the
  cron job's future `owner_id` requirement).
- **Migration filename timestamps** continue the existing
  `YYYYMMDDHHMMSS_description.sql` convention, sequential and after the
  last existing file (`20260622130000_rename_authenticated_rls_policies.sql`).
  This plan uses `20260623140000` through `20260623145500` in 5-minute

### Task 1: `split_taxonomy` table — create and seed

**Files:**
- Create: `supabase/migrations/20260623140000_create_split_taxonomy.sql`

**Interfaces:**
- Consumes: nothing from earlier tasks (first task).
- Produces: a `split_taxonomy` table with 4 seeded rows (`id` values
  `upper_lower`, `push_pull_legs`, `arnold`, `full_body`) — Task 5's
  `user_profile.preferred_split` column references this table's `id`
  column as a foreign key.

- [ ] **Step 1: Write the migration**

Create `supabase/migrations/20260623140000_create_split_taxonomy.sql`:

```sql
-- New in v2: training-split taxonomy. Drives user_profile.preferred_split
-- (FK added in a later migration) and recommendation_blocks.split_day_label
-- (free text, not FK'd to day_labels -- see
-- docs/superpowers/specs/2026-06-23-schema-v2-design.md Decision 12 for why
-- Postgres can't express an FK into an array element).
create table split_taxonomy (
  id text primary key,
  label text not null,
  day_labels text[] not null
);

alter table split_taxonomy enable row level security;

-- Read-only to authenticated app users (Settings-screen dropdown data);
-- no anon grant (unlike exercises) since this is behind the app's auth
-- gate, not linked from the public web dashboard. No write policy for
-- any role -- service role bypasses RLS for this seed and any future
-- admin reseed; no other role should ever write to taxonomy tables.
create policy authenticated_read_split_taxonomy on split_taxonomy
  for select
  to authenticated
  using (true);

insert into split_taxonomy (id, label, day_labels) values
  ('upper_lower', 'Upper / Lower', array['upper', 'lower']),
  ('push_pull_legs', 'Push / Pull / Legs', array['push', 'pull', 'legs']),
  ('arnold', 'Arnold Split', array['chest_back', 'shoulders_arms', 'legs']),
  ('full_body', 'Full Body', array['full_body']);
```

### Task 5: `user_profile` expansion — owner_id, pains rename/reshape, every new column

**Files:**
- Create: `supabase/migrations/20260623142000_expand_user_profile.sql`

**Interfaces:**
- Consumes: `split_taxonomy.id` (Task 1, FK target for `preferred_split`).
- Produces: `user_profile` with `owner_id` (backfilled, not null),
  `pains` (renamed/reshaped from `injury_constraints`), and 9 new
  columns. No later task in this plan reads these columns directly (they
  are inert until Phase 2+ engine/app code), but Task 12's RLS policy
  for `user_profile` depends on `owner_id` existing and being not-null
  here.

- [ ] **Step 1: Write the migration**

Create `supabase/migrations/20260623142000_expand_user_profile.sql`:

```sql
-- v2 user_profile expansion. Per
-- docs/superpowers/specs/2026-06-23-schema-v2-design.md Decisions 2 and 5:
-- owner_id is added nullable, backfilled to the one real auth user, then
-- set not null -- auth.uid() evaluates to null in a migration's execution
-- context so a bare `not null default auth.uid()` would fail outright
-- against the existing row. injury_constraints is renamed to pains and its
-- one existing row's flat {body_part: {active, note}} shape is rewritten
-- into the new jsonb-array-of-{body_part, severity, note, since} shape.

-- 1. owner_id: add nullable, backfill, then enforce not null + default.
alter table user_profile add column owner_id uuid references auth.users(id);

update user_profile
set owner_id = '89ac801b-5515-4388-80a7-03662030c487'
where owner_id is null;

alter table user_profile
  alter column owner_id set not null,
  alter column owner_id set default auth.uid();

-- 2. Rename injury_constraints -> pains, reshape the one existing row.
alter table user_profile rename column injury_constraints to pains;

update user_profile
set pains = '[
  {"body_part": "neck", "severity": 5, "note": "chronic stiffness, root cause treated as thoracic spine", "since": null},
  {"body_part": "ankles", "severity": 5, "note": "both injured over ~1.5yrs, highest reinjury risk under high volume / lateral pickleball movement", "since": null},
  {"body_part": "hips", "severity": 5, "note": "wants better mobility: deep squat, down dog, flexibility poses (also covers hamstrings)", "since": null},
  {"body_part": "shoulders", "severity": 5, "note": "right arm stronger and tighter/less mobile than left (right-side dominance)", "since": null}
]'::jsonb
where pains = '{
  "neck": {"note": "chronic stiffness, root cause treated as thoracic spine", "active": true},
  "ankles": {"note": "both injured over ~1.5yrs, highest reinjury risk under high volume / lateral pickleball movement", "active": true},
  "hips_hamstrings": {"note": "wants better mobility: deep squat, down dog, flexibility poses", "active": true},
  "right_dominance": {"note": "right arm stronger and tighter/less mobile than left", "active": true}
}'::jsonb;

-- jsonb arrays default to empty, not the old object default -- future
-- rows (a second real user, someday) start with no pains recorded.
alter table user_profile alter column pains set default '[]'::jsonb;

-- 3. Every other new v2 column, all nullable or defaulted.
alter table user_profile
  add column activities jsonb not null default '[]'::jsonb,
  add column preferred_split text references split_taxonomy(id) default 'upper_lower',
  add column current_goals jsonb not null default '[]'::jsonb,
  add column training_frequency_mode text not null default 'auto'
    check (training_frequency_mode in ('manual', 'auto')),
  add column training_frequency_manual jsonb,
  add column diet_preference text,
  add column weight_kg numeric(5, 2),
  add column birth_date date,
  add column location jsonb,
  add column healthkit_sync_enabled boolean not null default false;
```

```sql
-- user_profile: previously had no anon/authenticated policy at all
-- (service-role-only). Now also readable/writable by its own owner --
-- the mobile app's Settings screen (a later phase) needs the
-- authenticated user to read and update their own profile directly.
create policy owner_read_write_user_profile on user_profile
  for all
  to authenticated
  using (owner_id = auth.uid())
  with check (owner_id = auth.uid());

create policy owner_read_write_recovery on recovery
  for all
  to authenticated
  using (owner_id = auth.uid())
  with check (owner_id = auth.uid());

create policy owner_read_write_activity on activity
  for all
  to authenticated
  using (owner_id = auth.uid())
  with check (owner_id = auth.uid());

create policy owner_read_write_sessions on sessions
  for all
  to authenticated
  using (owner_id = auth.uid())
  with check (owner_id = auth.uid());

-- recommendations: v1 only granted authenticated read (never write --
-- the engine/cron is the only writer, via service role). v2 keeps that
-- asymmetry: owners can read their own recommendations directly, but
-- only the service role can write them (no with-check / for-all policy
-- here, select-only).
create policy owner_read_recommendations on recommendations
  for select
  to authenticated
  using (owner_id = auth.uid());

create policy owner_read_write_exercise_logs on exercise_logs
  for all
  to authenticated
  using (owner_id = auth.uid())
  with check (owner_id = auth.uid());

create policy owner_read_write_daily_feedback on daily_feedback
  for all
  to authenticated
  using (owner_id = auth.uid())
  with check (owner_id = auth.uid());
```

- [ ] **Step 4: Add transitive-ownership policies for the child tables with no owner_id of their own**

Append:

```sql
-- 4. recommendation_blocks/recommendation_block_exercises have no
-- owner_id of their own -- ownership is transitive through
-- recommendations.owner_id (one level for blocks, two levels for
-- block_exercises). Read-only to authenticated, matching
-- recommendations' own read-only-to-owner policy above -- only the
-- service role (Phase 2's program_builder.py) writes these.
create policy owner_read_recommendation_blocks on recommendation_blocks
  for select
  to authenticated
  using (
    exists (
      select 1 from recommendations r
      where r.id = recommendation_blocks.recommendation_id
        and r.owner_id = auth.uid()
    )
  );

create policy owner_read_recommendation_block_exercises on recommendation_block_exercises
  for select
  to authenticated
  using (
    exists (
      select 1 from recommendation_blocks b
      join recommendations r on r.id = b.recommendation_id
      where b.id = recommendation_block_exercises.block_id
        and r.owner_id = auth.uid()
    )
  );
```
