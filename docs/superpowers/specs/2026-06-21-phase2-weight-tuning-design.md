# Phase 2 — Weight-Tuning Prototype Design

## Context

Phase 0/1 (scaffolding + schema) are complete: the repo is public, Supabase is linked, and the v1 schema (`exercises`, `user_profile`, `recovery`, `recommendations` + `recommendations_public` view, `sessions`) is live with RLS verified. Per the roadmap in `docs/superpowers/specs/2026-06-20-bulletproof-architecture-design.md` section D, Phase 2 is next: prototype the deterministic scoring weights from `CLAUDE.md` against real historical data, in a notebook, before any engine code is written.

The original spec assumed a manual Oura CSV export for this phase, with the Oura API/PAT deferred to Phase 3. That assumption is revised here, based on discussion: Oura's API gives richer recovery *and* activity data than a manual export, and is the primary signal for "what activity happened" — not the Strong/Hevy export, which is being deferred to a later iteration (progressive-overload / weight-recommendation features, out of scope for v1). Pulling Oura history via API now also seeds the real `recovery` table and a new `activity` table, giving Phase 3 a head start without committing to production-grade ingestion code yet.

## Resolved decisions

- **Prototype format:** Python notebook (Jupyter), not a spreadsheet — lets the scoring logic be written once in Python and largely reused in Phase 3, consistent with the project's existing Python-engine decision.
- **Primary activity-load data source:** Oura API (recovery + daily activity + auto-detected workouts), not Strong/Hevy. Strong's export exists and is usable later, but only for a future iteration where the engine recommends weights/reps (progressive overload) — out of scope for this phase.
- **Oura auth method:** Personal Access Token (PAT), generated by Sohan at cloud.ouraring.com/personal-access-tokens, stored as `OURA_PERSONAL_ACCESS_TOKEN` in `.env` (gitignored, never committed). OAuth2 (client ID/secret, 3-legged flow) was considered and rejected — it's built for apps acting on behalf of *other* users' accounts, which doesn't apply to this single-user, no-auth-in-v1 project.
- **Historical window:** as far back as Sohan's Oura history goes (not capped at 30 days) — more data points to sanity-check weights against, at the cost of a larger first pull.
- **Pull cadence:** one-time historical backfill only. The script should be idempotent (safe to re-run / upsert-on-date) as a side effect of reasonable engineering, but it is explicitly NOT wired to any cron or daily-rerun workflow — that's Phase 4's job, done properly with GitHub Actions.
- **Data destination:** Supabase, not a local CSV. Recovery data (sleep, HRV, resting HR, readiness) lands in the existing `recovery` table. Activity data (activity score, calories, steps, detected workouts) lands in a **new** `activity` table — schema deferred until after an exploration step reveals the real Oura API response shape (see Sequence, step 2). Same RLS pattern as `recovery`: enabled, zero anon/authenticated policies, service-role-only.
- **Session-type ground truth:** for the historical window, reconstruct what session type (gym upper/lower, pickleball, run, rest, mobility) happened each day by combining the Strong export (gym days) with Oura's activity/workout data (pickleball/run days, once we see what it actually surfaces), with Sohan filling any remaining gaps (mobility, rest, anything missed) from memory. This reconstructed history is written into the existing `sessions` table — it already exists to hold "ground truth of what was actually done," and backfilling it now means rotation features (days since rest, days since mobility, same-pattern-as-yesterday) are queryable immediately rather than re-derived ad hoc in the notebook.
- **Code location:** the Oura pull/exploration code lives in `prototyping/weight-tuning/`, explicitly as throwaway/exploratory code — not `engine/`. Phase 3 is scoped to build the production Oura client (proper structure, retries, cron-ready, tested); keeping this version in `prototyping/` preserves that phase boundary rather than letting one-off exploration code masquerade as the real thing.

## Sequence

1. **Generate Oura PAT** (Sohan) — done; stored in `.env` as `OURA_PERSONAL_ACCESS_TOKEN`.
2. **Exploration script** (`prototyping/weight-tuning/oura_explore.py` or notebook cells) — hits Oura API v2 endpoints (`sleep`, `daily_readiness`, `daily_activity`, `workout`) for Sohan's account across full history, prints/dumps a handful of raw sample responses. No schema or table changes yet — purely to learn the actual field shapes before committing to columns.
3. **Design + migrate the `activity` table** — new Supabase migration (`supabase/migrations/<timestamp>_create_activity.sql`), columns chosen from what step 2 actually returned (expected shape: date, activity_score, total_calories, active_calories, steps, workout count/types — finalized after exploration). RLS: enabled, no anon/authenticated policy (same sensitivity class as `recovery`).
4. **Historical backfill script** (`prototyping/weight-tuning/oura_pull.py`) — pulls Sohan's full Oura history (sleep + readiness + activity + workouts), upserts into `recovery` and `activity` via the service role. One-off run; idempotent but not cron-wired.
5. **Session-history reconstruction** — parse the Strong export for gym days, cross-reference Oura activity/workout data for pickleball/run days, Sohan fills remaining gaps from memory. Write the resulting day-by-day session history into the existing `sessions` table (service role; `recommendation_id` left null since no recommendation existed historically).
6. **The notebook** (`prototyping/weight-tuning/weight_tuning.ipynb`) — loads `recovery` + `activity` + `sessions` from Supabase (via the existing `.env` credentials), implements the `CLAUDE.md` scoring algorithm (base rotation score, readiness gate, rest/mobility rotation weighting, same-pattern-as-yesterday penalty, pickleball/run gating rules, 10-day target ratio balancing) as Python functions with named, tunable weight constants, runs it day-by-day across the historical window, and displays each day's top pick + runner-up + score breakdown next to what Sohan actually did — so weights can be adjusted by inspection until the output feels right.

## Out of scope (this phase)

- Strong/Hevy as a primary data source (deferred to a later progressive-overload iteration)
- OAuth2 Oura integration
- Daily/cron automation of the Oura pull (Phase 4)
- Production-grade Oura client code, tests, error handling (Phase 3)
- Any change to `recommendations` or its public view

## Verification approach

- Step 2: exploration output reviewed by Sohan/Claude together — confirms what fields are actually available before the `activity` table is designed.
- Step 3: `supabase db push` applies cleanly; service-role insert + anon-blocked check via curl, mirroring the pattern used for `recovery` in Phase 1.
- Step 4: row counts in `recovery`/`activity` after the backfill match the expected number of days in Sohan's Oura history (spot-checked, not necessarily exact given gaps in ring-wearing).
- Step 5: `sessions` row count and spot-checked dates reviewed by Sohan against his own memory.
- Step 6: notebook output reviewed by Sohan — "does this look like what I should have been told to do on these real days?" — the same sanity-check bar as the original Phase 2 plan.
