# Autonomous build pipeline — design spec

## Background

After approving the mobile interface pivot
(`2026-06-22-mobile-interface-design.md`) and its bootstrap plan
(`2026-06-22-mobile-app-bootstrap.md`), the user asked for a way to drive
the *entire remaining roadmap* — not just the mobile app, but engine
productionization, the daily cron, and the eventual web dashboard — through
an autonomous multi-agent pipeline modeled on five roles: a planning agent,
a developer agent, a testing agent, a critique agent, and a reporting
agent, working continuously without per-step input from him.

This spec is about the *pipeline itself* — the mechanism, the roles, the
autonomy boundaries, and the execution order — not about the app features
it will build. Those stay scoped to their own specs/plans (the existing
mobile interface design, plus new ones the Planner role generates for the
phases that don't have one yet).

## Goals

- Drive the full remaining roadmap (mobile bootstrap completion → engine
  productionization → daily cron → HealthKit sync → recommendation UI →
  web dashboard) to completion in one continuous run, without the user
  answering clarifying questions mid-run.
- Map the user's five described roles onto concrete, dispatchable units of
  work using this harness's actual tool surface (the Agent tool's fixed
  subagent types), rather than anything requiring new infrastructure.
- Reuse and extend the two-tier review pattern already proven in this
  project (`.superpowers/sdd/` per-task reports + a final whole-branch
  review), rather than inventing a parallel reporting system.
- Give the user genuine visibility into what happened, in plain language,
  without him having to read every subagent's raw output.

## Non-goals

- Building new infrastructure (no new "agent type" registration, no new
  services). The pipeline is a *procedure* the orchestrating session
  follows using existing tools (Agent tool, git, the existing skills).
- Running across multiple sessions/days unattended (rejected in favor of a
  single long-running session — see Decisions below).
- Pausing for human review on git operations or TestFlight submissions
  (explicitly rejected by the user — see Decisions below).

## Decisions

### Scope: full remaining roadmap, not a single trial phase

The user chose to run this across the entire backlog immediately rather
than validating the pipeline on one phase first. Risk accepted explicitly:
this means the first real exercise of the pipeline doubles as a full-scale
run, not a contained pilot.

### Backlog order

Dependency-driven, not arbitrary — recommendation data needs to exist
before there's anything real for the UI to render, hence engine work comes
before the HealthKit/UI phases:

1. Finish the mobile bootstrap plan (`2026-06-22-mobile-app-bootstrap.md`,
   Tasks 2–8 — Task 1 is already done).
2. Engine productionization (roadmap Phase 3 — `engine/run_daily.py`,
   built on `prototyping/weight-tuning/scoring.py`).
3. GitHub Actions daily cron (roadmap Phase 4) — runs the engine each
   morning, writes to `recommendations`.
4. HealthKit → Supabase sync (mobile feature — the actual fix for the
   Apple Watch workout gap that started this whole pivot).
5. Recommendation/summary UI in the mobile app (now has real data from
   steps 2–3 to render against).
6. Web dashboard (roadmap Phase 5 / "Phase B" in the interface spec) —
   public, read-only, built on the existing `recommendations_public` view.

Phases without an existing spec/plan (2, 3, 5, 6) get one generated by the
Planner role as part of the pipeline run — see Roles below.

### Roles → mechanism

This harness has a fixed set of subagent types (`general-purpose`,
`Explore`, `Plan`, `claude`, etc.) — there's no way to register new
persistent "agent types." Each of the user's five roles is therefore the
orchestrating session (this conversation, running continuously) dispatching
a subagent via the Agent tool with a role-specific prompt:

- **Planner** — a `Plan`-type subagent per backlog phase. Produces a
  `writing-plans`-format implementation plan (exact file paths, code,
  steps — same shape as the existing mobile bootstrap plan), saved to
  `docs/superpowers/plans/`. If the phase has no spec yet, the Planner
  writes a short one first (following the brainstorming spec template,
  minus the back-and-forth — see "No mid-run questions" below).
- **Developer** — a `general-purpose` subagent implementing one task from
  the active plan, TDD-style, following this repo's existing conventions
  (small focused files, no unrequested abstraction, tests first).
- **Tester** — a `general-purpose` subagent that independently verifies the
  Developer's work: runs the test suite, tries to break it with cases the
  Developer might not have covered, and checks the task's actual stated
  acceptance criteria rather than just "tests are green."
- **Critic** — a `general-purpose` subagent reviewing code quality,
  simplification opportunities, and consistency with the project's stated
  voice (CLAUDE.md's Bryan Johnson/Blueprint framing, "no over-engineering"
  principle). Runs twice per phase: lightly per-task, and once more as a
  full whole-branch pass before merge — mirroring the two-tier pattern
  already used successfully in `.superpowers/sdd/progress.md` for Phase 0/1
  and Phase 2.
- **Reporter** — after each phase completes (merged, skipped, or partially
  stuck), summarizes what shipped, what was skipped and why, and what's
  next. Posts this to the user in chat and appends it to
  `docs/superpowers/reports/autonomous-build-log.md`. Per-task technical
  detail keeps living in `.superpowers/sdd/` (briefs, reports, the
  `progress.md` log) exactly as it already does — the new log is the
  human-readable rollup, not a replacement.

### Autonomy boundaries

Explicitly approved to run with **no human checkpoint**:
- Local git commits.
- Pushing to remote / merging to master.
- TestFlight / App Store Connect submissions.

Explicitly requires pausing and asking the user:
- Anything that could incur or increase a cost (triggering a paid EAS
  build, changing a Supabase plan tier, enrolling in any other paid
  service, etc.).

Safety net added without reducing autonomy: each phase works on its own
branch, merged to master automatically once it clears testing + critique.
This costs nothing in terms of the "no human checkpoint" requirement — it
just means a failed phase doesn't land partially-broken on master.

### Revision loop

A failing test or a critique rejection sends the task back to the
Developer role, capped at **3 revision rounds**. If still failing after 3
rounds: **skip and continue** — flag it clearly in that phase's report,
leave its branch unmerged, and move on to the next task/phase regardless of
whether anything downstream depended on it. The pipeline never pauses on
its own initiative; only a money-spend decision does.

### No mid-run questions

Every dispatched subagent (Planner included) is explicitly instructed never
to pause and ask the user a clarifying question. If a role hits genuine
ambiguity, it makes the most reasonable call itself and documents the
assumption in its output/report, rather than blocking. This is a real
behavior difference from how the orchestrating session normally operates
(which does ask the user one question at a time during brainstorming) —
the dispatched subagents do not get that luxury, by the user's explicit
choice.

### Run mechanism

A single long-running session: the orchestrating session (this
conversation) chains Agent-tool dispatches back-to-back across every task
in the backlog, without waiting on the user between dispatches, until the
backlog is exhausted or a money-spend decision is hit. This stops if the
session is closed — explicitly chosen over a persistent
scheduled/cross-session mechanism, trading "survives you closing the
laptop" for "easier to watch live."

## Expected cost/scale (informational, not a design constraint)

Across 6 phases, each with several tasks and up to 3 possible revision
rounds per task, this is realistically 80-150+ subagent dispatches in one
continuous run — meaningful real time and token cost. The user was made
aware of this before approving the design.

## Out of scope for this spec

- The actual content of phases 2, 3, 5, 6's specs/plans — those are
  generated at run time by the Planner role, per phase, not written here.
- Android support, multi-user support — both already out of scope per the
  mobile interface design spec and CLAUDE.md's standing principles.
