Important findings (should fix before or shortly after merge):
- `App.tsx`'s new HealthKit `useEffect` (lines 21-36) fires unconditionally on launch and every foreground transition, with no gate on `session` being non-null. `activity`'s RLS (`supabase/migrations/20260622130000_rename_authenticated_rls_policies.sql`) grants `for all using (true) with check (true)` only to the `authenticated` role -- there is no `anon` policy on `activity` at all (the original migration's own comment: "Deliberately no anon/authenticated policies: daily activity detail is as personal as recovery data"). So on a fresh install before the user has tapped "Sign in with Apple" (or any time the Supabase session has expired), `syncHealthKitWorkouts()` still runs: it triggers the real iOS HealthKit permission prompt before the user has authenticated into the app at all, then the `supabase.from('activity').upsert(...)` call gets rejected by RLS and is swallowed by the existing `console.warn`-and-return path. Not a crash and not a security hole (RLS is correctly doing its job rejecting the unauthenticated write), but it's a real, observable bad first-run experience -- the HealthKit permission sheet is a one-shot system UI surface, and burning it pre-sign-in on a fresh TestFlight install means the "real" permission grant that matters (post-sign-in, when sync can actually persist) has already been spent. Fix: gate the sync `useEffect` on `session` being non-null (re-run it, or move it to fire from the auth-state-change effect once `session` is set, instead of an independent empty-deps effect).
